Exploit!

File

首先透過 /query?site=URL 可以發現使用 http:// 和 file:// 可以達成 SSRF 和任意讀檔。

一開始我主要都在用 file:// 找線索其中 /etc/hosts 可以知道機器為 GCP 主機。

接下來透過 /proc/self/environ 得知 working directory 為 /opt/workdir

當時也透過 /proc/self/cmdline 得到 server source code 檔名並得到 source code

HTTP

隊友在發現主機為 GCP 之後就嘗試往 GCP 打，首先我們要先拿到 token。
GCP metadata API Docs 傳送門

取得 token 的 api 為 http://35.194.175.80:8000/query?site=http://metadata.google.internal/computeMetadata/v1/instance/service-accounts/default/token

在使用 internal API 時需要在 header 放上 Metadata-Flavor: Google，也因為先前看到 source code，其中使用了 urllib.request.urlopen，而這個函式是可以達成 header injection Source，所以最後的 payload 會長這樣

另外，我們可以透過 http://metadata.google.internal/computeMetadata/v1/project/project-id 來取得專案 ID

所以我們得到 project-id: balsn-ctf-2020-tpc。

之後我們因為在 compute 並沒有摸到奇怪的東西，所以隊友開始朝向 storage 摸。
GCP Storage API Docs
以下是他的思路

GCP Storage

首先我們要先找到 project 底下有什麼 buckets，那在打 API 的時候要加上在前面抓到的 access_token。
API Docs 傳送門

可以知道在 project balsn-ctf-2020-tpc 下有一個 bucket 叫做 asia.artifacts.balsn-ctf-2020-tpc.appspot.com

那我們就可以透過 API 找到 bucket 底下的 objects
API Docs 傳送門

而 mediaLink 所回傳的就是 object 檔案，格式為 tar.gz，我這邊可以寫個小工具把全部抓下來

最後再把全部 object 解壓縮，我這邊直接寫一個 shell script for i in $(ls | grep "tar.gz"); do tar zxvf $i; done

那最後就可以看到其實這是一個 disk 的備份，只是切成了好幾份

GET FLAG!

最後就可以在 /opt/workdir 中找到 flag 了 (flag-6ba72dc9ffb518f5bcd92eee.txt)

FLAG: BALSN{What_permissions_does_the_service_account_need}
